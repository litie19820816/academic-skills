# Entity Detection — 实体检测

生产级代理应在**每条入站消息**上触发实体检测。这是使知识脑随时间复合的信号检测循环。

## 核心原则

**不要阻塞对话。** 实体检测和富集应与响应并行发生，用户不应等待脑写入才得到答案。

---

## 检测协议（每次消息都执行）

1. **扫描消息**中的实体提及：人、公司、概念、原创思考。触发于每条消息（纯操作性的除外）。

2. **对每个检测到的实体：**
   - `search_files` 在脑目录中搜索该名称 — 档案是否已存在？
   - **如果存在：** 加载上下文，用已有综合真相告知响应。如果消息包含新信息，更新档案。
   - **如果不存在：** 评估可关注度（见下方）。如果值得追踪，用适当类型和 slug 创建新档案。

3. **创建或更新档案后：**
   ```bash
   # 如果使用文件存储，同步到脑目录
   git add ~/.hermes/brain/ && git commit -m "Entity update: {entity-name}"
   ```

4. **并行于响应，不要等待。** 用户先得到答案，脑写入在后台发生。

---

## 什么算值得关注的实体

### 高价值（最高优先级）

- **用户的原创思考**（想法、论点、观察）—— 最高价值信号。精确捕获原话——用户的语言本身就是洞察。不要改写。
  - 新颖的观察或论点
  - 框架、心智模型、启发法
  - 用户注意到的、别人忽略的连接
  - 有推理的反主流立场
  - 对外部刺激的强烈反应（什么触发了，为什么）

- **与用户互动的人** — 同事、合作者、行业人士、编辑、出版商等
- **与用户工作相关的公司** — 正在合作、考虑合作、竞争对手、所在领域公司
- **用户正在做的项目** — 副业、课程、内容创作、书籍

### 中等价值

- **行业概念和框架** — 用户讨论但尚未内化的思维模型
- **媒体内容的主要主题** — 文章/视频/播客的核心论点
- **行业动态** — 趋势、事件、变化

### 不建档

- 随机提及无关系信号
- 纯粹操作性的指令（"帮我订个会议室"）
- 用户明确说只是随口一提不想追踪的内容

## 实体类型和处理

| 类型 | 目录 | 模板参考 |
|------|------|----------|
| `person` | `people/` | 人物模板 |
| `company` | `companies/` | 公司模板 |
| `concept` | `concepts/` | 概念模板 |
| `project` | `projects/` | 项目模板 |
| `media` | `media/{type}/` | 媒体模板 |

---

## 冲突处理

- **用户说的 vs. 脑里存的了** — 用户最新说的优先。更新档案，注明矛盾。
- **用户否认脑里存的信息** — 立即标记为 `[Source: User, denied, YYYY-MM-DD]`，不删除原信息。
- **用户纠正一个具体事实** — 只改那个具体事实，其他保持不变。

---

## 执行检查清单

每次实体检测完成后，确认：
- [ ] 所有检测到的实体都在脑中有记录或已建档
- [ ] 新信息已写入相关档案的时间线
- [ ] 相关实体的反向链接已更新
- [ ] 所有写入都有 `[Source: User, ...]` 引用
- [ ] 脑已同步（git commit）

## 工具

- `search_files` — 在脑目录中搜索实体名
- `read_file` — 读取现有档案
- `write_file` — 创建/更新档案
- `terminal` — git 同步
