# Requisite Variety 必要多样性

## 核心原理

罗斯·艾什比（W. Ross Ashby）提出的必要多样性定律（Law of Requisite Variety）：

```
一个控制器要有效控制一个系统，
其自身的多样性必须 ≥ 被控系统的多样性

换句话说：
你的方案必须足够"复杂"来应对问题的"复杂性"

如果问题有5个维度，但你的方案只能处理2个维度，
那么另外3个维度的变化会绕过你的控制
```

**这是系统设计的黄金定律：复杂度匹配。**

## 触发检测

AI在以下语境中必须主动激活本skill：

- 组织设计、产品架构、风险管理
- 制定方案/策略时问"这个方案够用吗"
- 评估某个系统/机制是否能有效解决问题
- "为什么这个方法没有效果"
- "为什么政策执行总是走样"
- 制度设计、流程设计
- 评估投资组合、应对复杂局面

## 核心概念

```
多样性（Variety）= 可能状态的数量

一个骰子：6种可能状态 → 多样性 = 6
一个硬币：2种可能状态 → 多样性 = 2

一个控制系统：
- 必须能区分足够多的系统状态
- 必须有足够多的应对手段
- 才能实现有效控制
```

## 分析流程

### 第一步：识别问题的复杂度维度

```
问题复杂度分解：

问题：[具体描述]

复杂度维度清单：
  维度1：[什么维度的变化/不确定性]
    - 可能状态数量：[列出各种可能]
    - 每种状态的概率/频率：[如果可以估计]
    
  维度2：[…]
  维度3：[…]

总复杂度评估：
  - 核心维度：[最重要的N个维度]
  - 变异来源：[每个维度上的变化来自哪里]
  - 已知 vs 未知：哪些维度已经被理解，哪些还是黑箱
```

### 第二步：评估现有方案的维度覆盖

```
现有方案：[描述你的方案/策略/系统]

方案复杂度分解：
  维度覆盖检查：
    问题维度1 → 方案有应对：[是/否] → 应对方式：[描述]
    问题维度2 → 方案有应对：[是/否] → 应对方式：[描述]
    问题维度3 → 方案有应对：[是/否] → 应对方式：[描述]
    …

方案覆盖盲区：
  维度1 → 方案无法处理 → 失控风险：[…]
  维度2 → 方案无法处理 → 失控风险：[…]
```

### 第三步：评估方案与问题的复杂度匹配度

```
匹配度评估：

覆盖维度数：N / M（M=问题总维度数）
覆盖质量：
  - 完全覆盖：[N个维度都有充分应对]
  - 部分覆盖：[N个维度，但某些维度应对不充分]
  - 严重不足：[只有表层维度被覆盖，深层维度完全缺失]

失控风险评估：
  - 高风险：[未覆盖的维度可能导致系统完全失效]
  - 中风险：[未覆盖的维度会降低系统效果]
  - 低风险：[未覆盖的维度影响有限]
```

### 第四步：设计复杂度补充方案

```
复杂度补充建议：

方案A：扩大方案维度覆盖
  - 补充应对：[哪个维度需要新增处理能力]
  - 具体建议：[如何增强方案]
  - 成本/收益：[补充的代价和效果]

方案B：降低问题复杂度
  - 策略：[是否可以简化/分解问题，减少需要应对的维度]
  - 前提：[什么条件下简化是可行的]

方案C：接受不可控维度
  - 策略：[明确放弃对某些维度的控制，转而监控]
  - 风险预警机制：[当不可控维度出现异常时如何响应]
```

## 输出格式

```
【必要多样性分析】

问题界定：
  问题描述：[…]
  复杂度维度：共M个

问题维度分解：
  维度1：[名称] - 可能状态：[…]
  维度2：[名称] - 可能状态：[…]

方案维度覆盖：
  已覆盖维度：[N个，包括哪个维度、应对方式]
  未覆盖维度：[X个，包括哪个维度、失控风险]

复杂度匹配评估：
  匹配度：[完全覆盖/部分覆盖/严重不足]
  核心风险：[未覆盖维度中最危险的那个]

补充建议：
  [针对每个未覆盖维度的具体补充建议]
```

## 经典应用场景

**场景1：组织设计**
```
问题：如何让团队持续创新？
维度分析：
  维度1：人员能力差异 → 覆盖：招聘/培训
  维度2：激励机制 → 覆盖：OKR/奖金
  维度3：心理安全感 → 覆盖不足 ← 关键缺失
  维度4：信息流通路径 → 覆盖不足 ← 关键缺失

结论：只覆盖前两个维度不够，必须同时建立心理安全感和信息流通机制
```

**场景2：产品风险管理**
```
问题：如何防止产品重大事故？
维度分析：
  维度1：技术风险 → 覆盖：代码审查/测试
  维度2：运维风险 → 覆盖：监控/告警
  维度3：人为失误 → 覆盖不足 ← 关键缺失
  维度4：外部攻击 → 覆盖不足 ← 关键缺失

结论：技术和运维维度覆盖良好，但人为失误和外部攻击维度是主要漏洞
```

## 注意事项

- **维度分解要全面**：常见的错误是只分解表层维度，遗漏深层维度
- **不是所有维度都能控制**：接受某些维度不可控，比假装可控更诚实
- **过度的复杂度也有代价**：不是维度越多越好，过度复杂的控制方案本身会成为问题
- **匹配而非碾压**：方案复杂度 = 问题复杂度，不需要过度碾压
